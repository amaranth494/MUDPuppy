## 1. Executive Summary

PR01 expands the automation system beyond the MVP's rule-based pattern→response model to support conditional and time-based automation logic. This significantly increases the power available to users while maintaining the safety boundaries established in the MVP automation engine.

PR01 is the first Pre-Release specification, representing the transition from MVP stability to feature expansion while preserving architectural principles.

---

## 2. Problem Statement

The MVP automation model supports only simple pattern matching with direct command responses:

```
Trigger: "You are hungry"
Action: eat bread
```

This limitation prevents players from creating sophisticated automation behaviors that respond to game state. Players cannot:

- Make decisions based on character stats (health, mana, gold)
- Schedule delayed or repeating actions
- Create dynamic aliases that adapt to context
- Build automation that mimics advanced desktop MUD clients

PR01 addresses these gaps while maintaining the safety guarantees that protect users from automation loops and abuse.

---

## 3. Objectives

PR01 must deliver:

1. **Automation Logic Engine** — Conditional execution with IF/ELSE blocks, comparison operators, logical operators, and variable substitution
2. **Automation Function Syntax** — Reserved prefix `#` for logic commands (#IF, #ELSE, #ENDIF, #SET, #TIMER, #ENDTIMER, #CANCEL)
3. **Variable Operations** — Programmatic variable manipulation via #SET command, unified resolver, persistent write-through, system variable protection, session variables
4. **Timer System** — Delayed and repeating automation actions
5. **Integration** — Logic and timers work within triggers, aliases, and future automation scripts
6. **Safety Controls** — Timer limits, evaluation timeouts
7. **UI Enhancements** — Multi-line editing, indentation support, syntax validation
8. **Variable UI Sync** — Real-time update of variables panel when #SET changes profile variables
9. **Help Documentation** — Update help content to explain variable types and #SET

PR01 does NOT include:

- Full scripting language or runtime
- Automation debugger
- Script import/export
- Trigger chains

> **This is an automation logic expansion spec, not a full scripting environment.**

## 5. Automation Logic Engine Specification

### 5.1 Conditional Syntax

PR01 introduces a lightweight logic layer to automation rules:

```
#if ${hp} < 30
    cast heal
#else
    attack goblin
#endif
```

### 5.2 Supported Features

| Feature | Description | Example |
|---------|-------------|---------|
| Conditional logic | IF / ELSE execution | `#IF ${hp} < 30` |
| Comparison operators | > < >= <= == | `${gold} >= 1000` |
| Logical operators | AND / OR | `${hp} < 50 AND ${mana} > 10` |
| Variable substitution | ${var} references | `${target}` |

### 5.3 Syntax Examples

**Simple IF:**
```
#if ${hp} < 30
    cast heal
#endif
```

**IF/ELSE:**
```
#if ${hp} < 30
    cast heal
#else
    attack goblin
#endif
```

**Complex Condition:**
```
#if ${hp} < 50 AND ${mana} > 10
    cast heal
#endif
```

**Variable Comparison:**
```
#if ${target} == dragon
    warning Dragon detected!
#endif
```

## 6. Automation Function Syntax Specification

### 6.1 Reserved Prefix

All automation functions use a reserved prefix `#` to distinguish them from normal MUD commands. This prevents collisions with standard game commands.

### 6.2 Supported Functions

| Function | Description |
|----------|-------------|
| `#IF` | Begin conditional block |
| `#ELSE` | Alternate condition branch |
| `#ENDIF` | End conditional block |
| `#SET` | Assign value to variable |
| `#TIMER` | Create delayed/repeating action |
| `#ENDTIMER` | End timer block |
| `#CANCEL` | Cancel a timer |

### 6.3 Function Reference

**#IF**
```
#if ${condition}
    command
#endif
```

**#ELSE**
```
#if ${condition}
    command1
#else
    command2
#endif
```

**#SET**
```
#SET variable_name value
```

**#TIMER (Delayed)**
```
#TIMER name 10s
    command
#ENDTIMER
```

**#TIMER (Repeating)**
```
#TIMER name 5s REPEAT
    command
#ENDTIMER
```

**#CANCEL**
```
#CANCEL timer_name
```

## 7. Variable Operations Specification

### 7.1 Variable Types

PR01 defines three distinct variable types:

| Type | Syntax | Scope | Persistence |
|------|--------|-------|-------------|
| Profile variables | `${variable}` | Connection profile | Persisted to database |
| Session variables | `%VARIABLE` | Session only | Cleared on disconnect |
| System variables | `%TIME`, `%CHARACTER` | Read-only | Built-in |

### 7.2 Variable Resolution Precedence

When resolving a variable reference, the system checks in this order:

1. **Session variables** (%1, %2) — Runtime context overrides saved config
2. **Profile variables** (${name}) — Saved user configuration
3. **System variables** (%TIME, %CHARACTER) — Read-only built-in values

This order allows runtime context to take precedence while preserving saved configuration.

### 7.3 Unified Variable Resolver

All automation subsystems MUST call the same resolver:

- Alias expansion
- Trigger actions
- Logic evaluator
- Timer execution

This prevents variable divergence across subsystems.

### 7.4 #SET Command (Persistent Write-Through)

Variables gain programmatic manipulation via the #SET command:

```
#SET target goblin

#if ${gold} > 1000
    bank deposit
#endif
```

When evaluator processes #SET:
1. Update runtime variable map
2. Persist change to profile storage via API
3. Update UI state immediately

This is a write-through operation. If backend persistence fails:
- The value remains set in runtime memory
- Automation continues without interruption
- The failure is logged for diagnostics

### 7.5 System Variable Protection

The resolver MUST reject attempts to modify system variables:

```
#SET %TIME = noon  // REJECTED - system variable
#SET %CHARACTER = newname  // REJECTED - system variable
```

A user-visible error must be returned.

### 7.6 Session Variables

Session variables (%1, %2) are temporary variables populated from alias arguments or trigger pattern captures. They live in the session map and are cleared when:

- The connection ends (disconnect)
- A new connection begins (connect)

```
#SET %1 goblin    // Session variable
#SET %TARGET orc  // Profile variable
```

Session variables use `%` prefix without braces and are not persisted.

## 8. Timer System Specification

### 8.1 Timer Types

PR01 introduces two timer types:

**Delayed Action (one-time):**
```
#TIMER heal 10s
    cast heal
#ENDTIMER
```
Executes `cast heal` once after 10 seconds.

**Repeating Timer:**
```
#TIMER regen 5s REPEAT
    cast regen
#ENDTIMER
```
Executes `cast regen` every 5 seconds until cancelled.

### 8.2 Timer Management

- Timers are identified by name
- Multiple timers can run simultaneously
- Timers can be cancelled via #CANCEL

**Cancel Timer:**
```
#CANCEL regen
```

### 8.3 Timer Syntax

```
#TIMER [name] [duration] [REPEAT]
    [commands]
#ENDTIMER
```

- **name**: Unique identifier for the timer
- **duration**: Time interval (supports s, m, h units)
- **REPEAT**: Optional flag for repeating timers

## 10. Safety Controls Specification

### 10.1 Existing MVP Protections

PR01 retains all MVP automation safeguards:

| Protection | Purpose |
|------------|---------|
| Recursion limits | Maximum alias expansion depth |
| Command dispatch limits | Maximum commands per dispatch |
| Trigger rate limiting | Minimum time between trigger firings |
| Automation circuit breaker | Automatic pause when loops detected |

### 10.2 New PR01 Protections

PR01 introduces additional safeguards:

| Protection | Purpose |
|------------|---------|
| Timer limit | Prevents timer floods (max 10 active timers) |
| Nested logic limit | Prevents runaway scripts (max 3 #IF depth) |
| Evaluation timeout | Backend-enforced timeout for all automation evaluation (max 500ms) |

**Evaluation Timeout Scope:**

The evaluation timeout is a **backend (service-side) enforcement** that applies to all automation paths requiring evaluation:

- **Aliases** — Pattern matching and expansion
- **Triggers** — Output matching and response execution  
- **Timers** — Scheduled command execution
- **Logic blocks** — #IF/#ELSE/#ENDIF conditional evaluation

This aligns with the Constitution's service-first architecture and server-side automation sandbox model.

### 10.3 Safety Boundaries

The automation sandbox maintains these guarantees:

- No arbitrary code execution
- No file system access
- No network requests beyond MUD server
- No memory manipulation
- Command rate limiting prevents abuse

## 17. Future Considerations

### 17.1 Explicit Variable Scope Support

Future PR phases should consider adding explicit variable scope support to automation syntax:

| Syntax | Scope | Example |
|--------|-------|----------|
| `target` or `${target}` | Profile variable | `#SET target goblin` |
| `%target` | Session variable | `#SET %target goblin` |

**Rationale:** As automation complexity grows, users benefit from clear distinction between persistent state and temporary/session state. This reduces accidental profile pollution and makes debugging easier.

**Notes:**
- Profile variables remain as the default (no prefix required)
- Session variables use `%` prefix
- System variables remain read-only

### 17.2 Automation Variable Inspector

Future specifications should include an automation variable inspector for advanced troubleshooting.

**Recommended Capabilities:**
- View current profile variables
- View current session variables
- Show reserved system variables
- Indicate which values are persistent vs temporary
- Optionally show recent changes for debugging

**Rationale:** Advanced automation becomes much easier to troubleshoot when users can inspect current variable values. This aligns with the Constitution's requirement that automation remain visible and inspectable to the user.

### 17.3 Timer Inspector / Visibility

Users will eventually want to see active timers for debugging and management.

**Suggested Capabilities:**
- Show active timers
- Display remaining time
- Show repeat status
- Allow manual stop/start

**Example UI:**
```
Active Timers
-------------
heal (45s remaining)
buff (repeat every 60s)
```

**Rationale:** Timer-based automation becomes much easier to debug when users can visually inspect running timers. This aligns with the Constitution's transparency requirement and supports power users managing complex automation.

### 17.4 Timer Editor UI

If timers are profile-backed automation objects, they should eventually have the same UI editor pattern used for aliases, triggers, and variables.

**Rationale:** Consistency in UI patterns reduces cognitive load. Users familiar with alias/trigger editors should find timer editing intuitive. This likely becomes a later PR specification.

### 17.5 Automation Safety Visibility

Currently timeout events log to console. Future implementations should surface these events to users for debugging automation scripts.

**Example Messages:**
- "Automation paused: evaluation timeout exceeded"
- "Automation paused: circuit breaker triggered"
- "Automation paused: loop detected"

**Rationale:** Users need visibility into why their automation stopped. This aligns with the Constitution's automation transparency requirement.

### 17.6 Automation Debug Tools

PR01's expanded automation capabilities make debugging tools even more valuable. Future specifications should consider:

- **Variable Inspector** — View current profile and session variables
- **Trigger Firing Log** — See which triggers matched and executed
- **Timer Inspector** — View active timers with remaining time and repeat status
- **Alias Expansion Preview** — Preview how an alias will expand before execution

**Rationale:** These tools help users understand and debug complex automation. They align with the Constitution's requirement that automation remain visible and debuggable.

## 14. Technical Implementation Notes

### 14.1 Parsing Strategy

The automation engine shall parse # commands before MUD command dispatch:

1. Parse automation action text
2. Evaluate #IF conditions against variables
3. Execute resulting commands (filtered by condition)
4. Handle #SET, #TIMER, #CANCEL specially

### 14.2 Variable Storage

Variables shall be stored:

- Backend: In profile data (existing profiles table)
- Frontend: In connection context (SessionContext)
- Sync: Variables persist via profile save

### 14.3 Timer Execution

Timers shall be managed:

- Backend: Timer registry per session
- Execution: Async scheduler (Go routine)
- Persistence: Timer state NOT persisted across sessions (session-only)
