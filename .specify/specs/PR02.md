## 1. Overview

PR02 establishes the Internal Command Module (ICM) as the single authoritative system for recognizing, parsing, validating, normalizing, and dispatching all internal command forms and operator-led references in MUDPuppy.

### 1.1 Problem Statement

MUDPuppy already has meaningful automation behavior, canonical submission pathways, and safety systems, but operator handling is conceptually spread across too many feature paths. SP05 created the first real automation engine with alias processing, variable substitution, trigger processing, command separation, loop protection, queue backpressure, and the circuit breaker. SP04 and SP06 then reinforced that alternative command-entry surfaces such as keybindings and history recall must still route through the same canonical command pathway rather than bypassing it. PR01 proved the next architectural need is centralized operator resolution before more advanced command logic expands further.

### 1.2 Constitutional Framing

The frontend may author, inspect, validate, and document automation behavior, but automation that interacts with live session state, parsed output, timers, triggers, command dispatch, or operational tooling is constitutionally backend responsibility.

Structured directives must remain explicit, deterministic, bounded, human-readable, and integrated into the existing automation engine and safety systems.

Internal commands may not bypass recursion limits, command dispatch limits, trigger cooldowns, queue backpressure, circuit breaker protections, authentication, tenant isolation, or observability requirements.

### 1.3 Core Objective

Build the ICM as the **backend-authoritative** command-processing layer for all operator-driven internal behavior, with a defined calling contract from both frontend and backend.

After the ICM exists, PR02 must:
- Register and migrate the known internal command families
- Prove they still work through the ICM
- Define a repeatable process for adding new commands
- Audit older code for ad hoc logic that should instead call the ICM
- Convert those call sites
- Document the new model
- QA the whole flow thoroughly

## 2. Architectural Definition

> **Constitutional Reconciliation:** The Constitution (Sections IV.0 and IV.a) states that automation objects execute within the client runtime, while automation that interacts with live session state is a backend responsibility. PR02 reconciles this as follows:

> **Backend (Authoritative):** The backend is authoritative for command semantics and for any command effect that touches session state, persistence, observability, safety enforcement, or operational behavior.

> **Frontend/Client (Limited):** Authoring, preview, validation messaging, help display, and presentation. Client-side runtime helpers may still perform non-authoritative preview, authoring support, and constitutionally permitted local execution mechanics, but may not define competing command semantics.

> The ICM serves as the single source of truth for operator semantics across the system. The client may assist with preview and authoring, but authoritative runtime evaluation and dispatch remain backend-owned wherever live session behavior is involved.

### 2.1 ICM Command Lifecycle

The ICM defines one internal command lifecycle:

```
ingress → tokenize/recognize → validate → normalize → resolve → dispatch/report
```

> **Constitutional Bridge:** This lifecycle is the ICM's internal processing model. When resolving commands at runtime, the ICM must preserve the constitutional execution order: input → alias expansion → variable substitution → logic evaluation → queue dispatch → trigger evaluation. The ICM lifecycle handles recognition and normalization, while the constitutional order governs when each phase executes in the broader automation pipeline.

That lifecycle must be the same authoritative model regardless of whether the command originated from:
- Typed command input
- Keybinding dispatch
- History replay
- Alias expansion
- Trigger action
- Timer action
- Internal operational directive
- Future service tooling

### 2.2 Authority Model

**Backend ICM Core (Authoritative):**
- Command registry and handler resolution
- Execution contracts for session-state commands
- Safety enforcement (recursion limits, circuit breaker, rate limits)
- Command dispatch and operational actions
- Logging and observability directives
- Persistence-aware command effects

**Frontend Adapter (Client/Consumer):**
- Editor validation
- Help text and examples
- Safe previews where appropriate
- Command submission routing
- User-facing error reporting

### 2.3 Shared Schema

Frontend and backend exchange commands through a shared schema:

| Field | Description |
|-------|-------------|
| `raw` | Original raw input |
| `normalized` | Canonical command form |
| `operator` | Detected operator family (#, @, %, $) |
| `command` | Command identifier |
| `args` | Parsed arguments |
| `context` | Execution context (preview, submission, automation, operational) |
| `result` | Execution result or error |

## 3. Known Command Families

PR02 explicitly accounts for these operator families:

| Family | Operators | Description |
|--------|-----------|-------------|
| Structured Directives | `#` | Automation control: `#IF`, `#ELSE`, `#ENDIF`, `#SET`, `#TIMER`, `#START`, `#STOP`, `#CHECK`, `#CANCEL`, `#LOG` |
| User Variables | `$` | Profile variables editable by users via `#SET`: `${name}` or `$name` syntax. These appear in the UX and can be modified with commands. |
| System Variables | `%` | Session-scoped variables with two forms: numeric (`%1`, `%2`, etc.) for temporary arguments, and named (`%TIME`, `%CHARACTER`, etc.) for read-only client data. Both exist for the session lifespan and are cleared on disconnect. |
| Alias References | `@` | Alias invocation: `@aliasname` - invoke existing alias |

### 3.1 Variable Operator Clarification

- **`$` (User Variables):** Profile variables that are user-editable via `#SET` commands. These persist across sessions and are visible/editable in the Variables panel UI.
- **`%` (System Variables):** Session-scoped variables that exist for the session lifespan and are cleared on disconnect. These have two forms:
  - **Numeric form (`%1`, `%2`, etc.):** Temporary argument placeholders, automatically populated during alias/trigger expansion
  - **Named form (`%TIME`, `%CHARACTER`, etc.):** Read-only client data derived from internal client state, not directly editable via commands

### 3.1.1 % Namespace Resolution Precedence

| Rule | Specification |
|------|---------------|
| **Recognition Order** | Numeric patterns (`%` followed by digits) are always resolved as numeric system variables. Named patterns are resolved against the registered system variable registry. |
| **Numeric Form** | `%<digits>` is always session-scoped and cleared on disconnect. |
| **Named Form** | `%<identifier>` is always a system variable and read-only. |
| **Collision Policy** | Named system variables take precedence over numeric forms when the name matches a registered system variable. Unknown names pass through literally. |
| **Error Behavior** | Malformed references (e.g., `%` alone with no digits or identifier) pass through literally to preserve backward compatibility. |

### 3.2 Official Structured Directives

The official structured directives already recognized by doctrine remain:
- `#IF` — Begin conditional block
- `#ELSE` — Alternate condition branch
- `#ENDIF` — End conditional block
- `#SET` — Assign value to variable
- `#TIMER` — Create or update timer definition
- `#START` — Start an existing timer
- `#STOP` — Stop an existing timer
- `#CHECK` — Inspect timer status
- `#CANCEL` — Delete timer definition

PR02 additionally introduces `#LOG` as a constitutionally permitted internal operational directive managed by the ICM.

### 3.3 #LOG Governance

The `#LOG` directive is a **developer-facing operational command** with the following specifications:

| Aspect | Specification |
|--------|--------------|
| **Visibility** | Developer/Operational only (not end-user facing in help) |
| **Execution Contexts** | Automation, Operational |
| **Permission Model** | Permission-gated (admin/developer role) or trusted backend operational context |
| **Output Format** | Structured JSON with timestamp, level, message, correlation ID |
| **Routing Destinations** | Backend logging service, observability platform |
| **Required Metadata** | session_id, user_id, timestamp, level, correlation_id |
| **Help Documentation** | Not surfaced in user-facing help |

This directive demonstrates the command-first consolidation model and serves as the flagship example for future operational commands.

## 4. ICM Component Design

### 4.1 Core Components

```
┌─────────────────────────────────────────────────────────────┐
│                    Internal Command Module                   │
├─────────────────────────────────────────────────────────────┤
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐    │
│  │  Recognizer  │→ │  Validator    │→ │  Normalizer  │    │
│  └──────────────┘  └──────────────┘  └──────────────┘    │
│         ↓                                       ↓           │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐    │
│  │   Resolver   │← │   Registry   │→ │  Dispatcher  │    │
│  └──────────────┘  └──────────────┘  └──────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 Component Responsibilities

| Component | Responsibility |
|-----------|----------------|
| **Recognizer** | Tokenize input, identify operator families, extract command candidates |
| **Validator** | Check syntax validity, validate arguments, enforce grammar rules |
| **Normalizer** | Convert to canonical form, resolve variables, escape literals |
| **Resolver** | Resolve variable references, lookup aliases, compute expressions |
| **Registry** | Store command definitions, handlers, metadata, safety hooks |
| **Dispatcher** | Route to appropriate handler, enforce safety, report results |

### 4.3 Execution Contexts

ICM commands execute in different contexts:

| Context | Description | Authority |
|---------|-------------|-----------|
| **Preview** | Local validation, help display, syntax check | Frontend-only |
| **Submission** | User command entry, keybinding, history | Frontend → Backend |
| **Automation** | Alias expansion, trigger action, timer fire | Backend-authoritative |
| **Operational** | Service tooling, admin commands | Backend-only |

## 5. Safety Integration

### 5.1 Required Safety Hooks

The ICM must integrate with existing constitutional safeguards:

- **Recursion limits** — Maximum alias expansion depth
- **Command dispatch limits** — Maximum commands per dispatch
- **Trigger rate limiting** — Minimum time between trigger firings
- **Automation circuit breaker** — Automatic pause when loops detected
- **Evaluation timeout** — Maximum execution time per command
- **Queue backpressure** — Command queue depth limits

### 5.2 Safety Enforcement Points

| Phase | Safety Check |
|-------|-------------|
| Validate | Syntax validation, argument limits |
| Normalize | Variable resolution limits |
| Resolve | Expression complexity limits |
| Dispatch | Circuit breaker, rate limits, quotas |

## 6. Escape, Literal, and Pass-Through Behavior

### 6.1 Escape Grammar

The ICM must handle escape sequences to allow literal representation of operator characters:
- `\#` — Literal `#` character
- `\@` — Literal `@` character
- `\$` — Literal `$` character
- `\%` — Literal `%` character

### 6.2 Literal Pass-Through Rules

- **Undefined variables** — Must pass through literally (e.g., `${undefined}` outputs as `${undefined}`)
- **Unknown operators** — Pass through to MUD server unchanged if not a reserved family
- **Non-operator input** — Pass through to MUD server unchanged

### 6.3 Unknown Operator Handling

**Rule 1 — Unknown operator family (not reserved):**
- If the operator character is not a known reserved family (#, $, %, @), pass through to MUD server unchanged
- This preserves compatibility with non-standard prefixes

**Rule 2 — Unknown command inside reserved family:**
- If the operator is a reserved family (#, $, %, @) but the command is not recognized, return a validation error
- Do NOT pass through to MUD server — this would create ambiguity in the most important reserved families
- Log for debugging if enabled

**Examples:**
- `&somecommand` → pass through (unknown family)
- `#WHATEVER` → validation error (# is reserved but WHATEVER is unknown)
- `@undefined` → validation error (@ is reserved but undefined alias)

## 7. Migration Philosophy

### 7.1 Audit Requirements

No code outside the ICM should remain the authoritative interpreter of operator semantics.

Old production code must be audited for behavior such as:
- Prefix checks
- Inline alias expansion
- Special-case variable replacement
- Local directive parsing
- One-off logging helpers that should become command-driven
- Submission variants that bypass the canonical command path

### 7.2 Migration Priorities

1. **First** — Current structured directives (#IF, #SET, #TIMER, etc.)
2. **Second** — User variable references ($)
3. **Third** — System variables (numeric %1, %2, etc.)
4. **Fourth** — System variables (named %TIME, %CHARACTER)
5. **Fifth** — Alias references (@)
6. **Sixth** — Logging helpers → #LOG directive

| Requirement | Description |
|--------------|-------------|

| Purpose | Clear statement of execution context |
| Input grammar | BNF or equivalent specification |

| Validation rules | Argument types, ranges, required fields |
| Safety hooks | Which safeguards apply |
| Persistence rules | Whether command affects profile/session state |
| Visibility | User-facing help, error messages |
| QA scenarios | Test cases for validation |

## 10. Acceptance Criteria

PR02 should not be considered complete unless:

- [ ] One authoritative ICM exists and is used as the official internal command layer
- [ ] Backend-authoritative ICM core exists and is not merely mirrored from frontend code
- [ ] Known operator families are registered there
- [ ] Frontend calls the ICM through defined contracts
- [ ] Backend calls the ICM through defined contracts
- [ ] Variable families (user/system) are clearly differentiated and migrated
- [ ] Escape/literal/unknown-operator behavior is documented and regression tested
- [ ] #LOG behavior is specified, implemented, and verified under the chosen visibility model
- [ ] Prior scattered implementations are materially reduced through explicit conversion work
- [ ] Current commands still behave correctly after migration
- [ ] A documented process exists for creating future commands
- [ ] User-facing docs and release notes are updated
- [ ] QA proves major command entry paths and regressions from the UI
- [ ] Ordinary MUD commands pass through unchanged when not classified as internal commands
- [ ] Backend unit/integration tests exist for ICM recognizer, validator, normalization, handler registration, and safety boundaries
