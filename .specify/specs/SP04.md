# SP04 — Per-Connection Profiles & Input Customization (Pre-MVP)

> **Spec ID:** SP04  
> **Status:** Draft  
> **Author:** Product Lead  
> **Branch Prefix:** sp04-connection-profiles  
> **Constitutional Reference:** Constitution VIII (Deployment Governance), Constitution X (Pre-Push Build Requirements)  

---

## 1. Problem Statement

SP03 established:

- Persistent application shell
- Stable session authority
- Modal workspace system
- Saved connections (CRUD)
- Server-authoritative lifecycle discipline

However, the system currently lacks per-connection behavioral customization. All MUD connections behave identically from an input and display perspective.

This prevents:

- Custom keybinding layouts per MUD
- Custom scrollback settings
- Input behavior tuning
- Future automation scaffolding
- A clear foundation for triggers, aliases, variables

SP04 introduces Profiles — a per-connection configuration layer that binds input behavior and UI settings to a specific saved MUD connection.

> **This is a foundational spec, not a feature explosion spec.**

---

## 2. Objectives

SP04 must deliver:

1. **One Profile per Saved Connection** (1:1 relationship)
2. **Server-persisted Profile storage**
3. **Keybinding engine** (minimal)
4. **Profile-based UI settings**
5. **Input interception layer**
6. **Profile management UI**
7. **Strict Git lifecycle closure discipline**

SP04 does NOT introduce automation logic, scripting engines, or trigger execution.

## 4. Feature Set

### 4.1 Profile Data Model

**Database Table:** `profiles`

**Fields:**

| Field | Type | Description |
|-------|------|-------------|
| `id` | UUID | Primary key |
| `user_id` | UUID | Foreign key → `users.id` |
| `connection_id` | UUID | Foreign key → `connections.id`, unique |
| `keybindings` | JSONB | Key mapping to commands |
| `settings` | JSONB | UI and behavior settings |
| `created_at` | TIMESTAMP | Creation timestamp |
| `updated_at` | TIMESTAMP | Last modified timestamp |

**Constraints:**

- `connection_id` must be UNIQUE (1:1 relationship)
- ON DELETE CASCADE from connections
- No secrets stored
- No executable code stored
- JSONB must validate to allowed structure (server-side)

**Profile Creation Rules:**

- When a saved connection is created → **automatically create default profile in same DB transaction**
- Connection creation endpoint must create profile within same transaction and rollback if profile insert fails
- If legacy connections exist without profile → migration script creates default profiles (see Section 4.6)
- Profile cannot exist without connection
- If profile creation fails, entire connection creation must be rolled back

### 4.6 Legacy Connections Migration

**Migration script must:**
- Insert profile rows for all existing connections in `connections` table
- Use default JSONB values (see Section 4.2)
- Run as part of migration 006
- Must complete before SP04 reaches staging merge

**PostgreSQL Extension Requirement:**

The SQL below uses `gen_random_uuid()` which requires the **pgcrypto** extension.

**Verification:** Ensure pgcrypto extension is installed in staging database before migration runs:
```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;
```

Alternatively, use application-generated UUIDs.

**SQL Migration Logic:**

```sql
INSERT INTO profiles (id, user_id, connection_id, keybindings, settings, created_at, updated_at)
SELECT 
  gen_random_uuid(),
  user_id,
  id,
  '{}'::jsonb,
  '{"scrollback_limit": 1000, "echo_input": false, "timestamp_input_echo": false, "word_wrap": true}'::jsonb,
  NOW(),
  NOW()
FROM connections
WHERE id NOT IN (SELECT connection_id FROM profiles WHERE connection_id IS NOT NULL);
```

### 4.2 Keybinding Engine (Minimal Viable Version)

**Goal:**

Allow mapping of key combinations to command strings.

**Example mappings:**

| Key | Command |
|-----|---------|
| F1 | `score` |
| Ctrl+1 | `cast heal` |
| Alt+W | `west` |

**Behavior Requirements:**

- Keybinding intercept occurs before normal input submission
- If keybinding exists → dispatch mapped command
- If no mapping → allow normal typing
- Must respect modal input lock (SP03)
- Must NOT execute while modal active
- Must NOT bypass rate limiting
- Must dispatch command through existing WebSocket data plane
- Must NOT send commands while disconnected
- **Must not auto-repeat on key hold** — keybinding fires once per discrete keydown event
- **Require keyup before re-trigger** (or implement keydown debounce)

**Keybinding Normalization:**

All key events must be normalized to a canonical format before lookup.

**Format:** `[Modifier+]*Key` (e.g., `Ctrl+Shift+F1`, `Alt+W`, `F1`)

Rules:
- Modifier order: Ctrl, Alt, Shift, Meta
- Case-insensitive key names
- Browser-reserved combinations may not work (document unsupported)
- OS-level shortcuts take precedence

**Collision Resolution:**

If user binds overlapping combinations (e.g., F1, Ctrl+F1, Shift+F1):
- Most specific match wins (longest modifier chain)
- Exact match required, no prefix matching

**Keybinding Execution While Input Has Focus:**

When user is typing in command input field:
- Keybindings SHOULD override normal typing
- F1 while typing dispatches the bound command
- This is intentional for quick access

**Dispatch Model:**

Keybinding execution must behave identically to:
> User types command → presses Enter

No alternate execution path allowed.

**Canonical Implementation Rule:**

There must be a **single function** responsible for command submission:
```typescript
submitCommand(source: 'typing' | 'keybinding', text: string)
```

Both:
- The `<Enter>` handler
- The keybinding handler

**Must route through this same function.**

Keybindings do not send commands directly. They invoke the same submission pathway used by typed commands.

This guarantees:
- Same trimming/normalization rules
- Same connection-state checks
- Same rate limiting behavior (server authoritative)
- Same local echo behavior
- Same command history behavior
- Same telemetry hooks (future-proofing)
- Same automation hook-in point (SP05+)

There must NOT be duplicated send logic in the terminal component.

**Storage Structure Example (JSONB):**

```json
{
  "F1": "score",
  "Ctrl+1": "cast heal",
  "Alt+W": "west"
}
```

**Default Profile JSON:**

```json
{
  "keybindings": {},
  "settings": {
    "scrollback_limit": 1000,
    "echo_input": false,
    "timestamp_input_echo": false,
    "word_wrap": true
  }
}
```

No null/undefined ambiguity permitted.

**Maximum Values:**

- Max keybindings per profile: **50**
- Max command length: **500** characters
- Max scrollback limit: **10000** lines
- Min scrollback limit: **100** lines

**JSONB Validation Rules:**

Server must validate:
- Reject unknown top-level fields in keybindings/settings
- Reject nested objects in keybindings
- Require string keys → string command values
- Enforce max 50 bindings
- Enforce max command length (500 chars)
- Enforce scrollback min/max (100–10000)
- Prevent schema drift inside JSONB

### 4.3 Profile Settings (Initial Scope)

`settings` JSONB may include:

| Setting | Type | Description |
|---------|------|-------------|
| `scrollback_limit` | integer | Terminal scrollback lines |
| `echo_input` | boolean | Local echo of typed commands |
| `timestamp_input_echo` | boolean | Prefix locally-echoed commands with timestamp |
| `word_wrap` | boolean | Toggle xterm wrap behavior |

> **Clarification (timestamp_input_echo):** This setting timestamps only locally-echoed commands (when `echo_input` is enabled). Server output remains untimestamped. This aligns with pre-MVP scope where timestamps are limited to user input feedback only.

**All settings must:**

- Be optional
- Have safe defaults
- Be validated server-side
- Be applied only when session connects

**Scrollback Limit:**

- Applied to xterm instance
- Must not cause memory instability
- Default: 1000 lines
- **Min: 100, Max: 10000**

**Echo Input:**

- If enabled: Locally echo command before server response
- If disabled: Wait for server response
- **Risk Note:** If MUD echoes commands server-side and local echo is enabled, duplicate command lines may appear. This is acceptable for MVP.

**Timestamp Input Echo:**

- If enabled: Prefix locally-echoed commands with timestamp
- Frontend-only rendering change
- Server output remains untimestamped
- **Implementation Note:** Must hook into input echo pipeline, prepend timestamp to line when echoing locally. Do NOT prefix raw ANSI stream text from server before xterm renders, as this may break color codes and escape sequences.

**Word Wrap:**

- Toggle xterm wrap behavior

### 4.4 Profile Fetch Timing

**Profile must be fetched when connection becomes active (on explicit connect), not on page load.**

**Required Sequencing (No Race Condition):**

```
User clicks Connect
    ↓
Fetch profile
    ↓
Apply profile (keybindings, settings)
    ↓
Then establish WebSocket connection
```

**Why This Order Matters:**

- If WebSocket connects first and profile lags, user may press F1 before keybindings loaded
- Keybinding not loaded = command not dispatched
- Must fetch/apply profile BEFORE WebSocket is ready for input

**Rationale:**
- Profiles are tied to connection
- Avoid stale caching
- Avoid prefetch overhead
- Avoid client-truth state

**Implementation:**
- On user click "Connect" → fetch profile → apply keybindings and settings
- Cache profile snapshot at connection start
- Ignore live profile updates until reconnect

**Refresh Behavior:**
- Browser refresh ends the active session
- User must explicitly reconnect after refresh
- Profile is loaded on connect
- Profile changes apply on next connect (not immediately)

**Failure Handling:**

If profile fetch fails → **connection must not proceed.**

| Error | Behavior |
|-------|----------|
| 404 (Not Found) | Block connect. Show user error: "Profile not found" |
| 500 (Server Error) | Block connect. Show user error: "Failed to load profile" |
| Network Error | Block connect. Show user error: "Connection failed" |

**Do not partially connect.** If profile cannot be loaded, the session must not establish. This preserves session determinism.

SP04 introduces a clean input abstraction:

```
Keyboard Event
       ↓
Profile Keybinding Check
       ↓
Command Dispatch or Default Behavior
       ↓
WebSocket send
```

**Requirements:**

- Deterministic
- No race conditions
- No command duplication
- No keystroke leakage
- Must survive resize and focus changes
- Must not interfere with ANSI rendering

### 4.5 Profile Management UI

Profile editing will occur via modal system (reuse SP03 modal container).

**Access Path:**

> Connections Hub → Edit → Profile Settings

**UI Capabilities:**

- View existing keybindings
- Add new keybinding
- Edit keybinding
- Delete keybinding
- Edit settings toggles
- Save profile

> **No advanced UX required. Minimal viable interface acceptable.**

**Behavior Rules:**

- Editing profile does NOT reconnect session
- Editing profile does NOT auto-apply mid-session
- **Profile changes apply on next connect only**
- **Cache profile snapshot at connection start**
- **Ignore live profile updates until reconnect**
- No background profile sync

## 6. Security Considerations

- Commands remain subject to server rate limiting
- **Keybinding dispatch does NOT implement client-side throttle**
- **Server rate limiting is authoritative**
- Profile JSON validated strictly
- Maximum binding count enforced (50)
- Maximum command length enforced (500)
- Maximum scrollback limit enforced (10000)
- No executable code stored
- No sandbox execution in SP04
- No dynamic evaluation

---

## 7. Risk Areas

### 7.1 Input Conflicts

- Browser-reserved key combos
- OS-level shortcuts
- IME interference

**Mitigation:**
- Document unsupported combinations
- Do not override system-critical keys

### 7.2 Modal Focus

Keybindings must not fire when modal active. Must reuse SP03 input lock discipline.

### 7.3 Scrollback Memory

Large scrollback limits may impact performance. Enforce max value.
