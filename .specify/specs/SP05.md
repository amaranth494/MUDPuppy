## 1. Problem Statement

SP04 established per-connection profiles with:
- Persistent settings (scrollback, echo, word wrap)
- Keybinding engine for keyboard shortcuts
- Profile-based UI configuration

However, users still manually type repetitive commands and cannot respond to server output automatically. This limits gameplay efficiency and prevents automation of common tasks.

SP05 introduces the first automation capabilities: **Aliases** and **Triggers**.

> **This is a foundational automation spec, not a scripting engine spec.**

> **Engineering Note:** The automation engine includes loop detection, command queue dispatching with backpressure, and circuit breaker protections to ensure runtime safety.

---

## 2. Objectives

SP05 must deliver:

1. **Alias system** for input command transformation
2. **Trigger system** for output-driven automation
3. **Environment variables** for reusable automation values
4. **Connection Settings UI** workspace for automation configuration
5. **Profile extension** with automation JSONB fields
6. **Server-side resource limits** for automation safety
7. **Client-side execution safeguards** to prevent runaway loops

SP05 does NOT introduce:
- Regex matching
- Multi-command triggers
- Timers
- Scripting engines
- Automation import/export

---

## 3. Non-Goals (Explicitly Out of Scope)

SP05 will NOT include:

- Regex-based triggers
- Multi-command trigger actions
- Timer-based automation
- Client-side scripting
- Automation import/export
- Automation sharing between users
- Server-side trigger execution (triggers execute client-side only)
- Output suppression ("gagging")
- Variable expressions or math
- Conditional logic in automation
- Trigger chains or nesting

> **SP05 establishes minimal automation scaffolding only.**

## 4. Architectural Principles

### 4.1 Single Command Pipeline

All automation must ultimately submit commands through the same pipeline used by manual input and keybindings.

```
user input / automation
        ?
submitCommand()
        ?
WebSocket
        ?
MUD server
```

**Rule:** Automation must never create alternate command execution paths.

### 4.2 Control Plane vs Data Plane

- **REST endpoints** continue to manage configuration and session control
- **WebSocket** continues to carry command submission and server output
- Automation must not bypass these responsibilities

### 4.3 Profile Persistence Model

Automation rules are stored as part of the existing per-connection profile.

```
Connection
   + Profile
       + settings
       + keybindings
       + aliases
       + triggers
       + variables
```

The UI will not expose the profile object directly. Users configure automation through **Connection Settings**.

### 5.2 Aliases

#### Definition

Aliases transform user-entered commands into different commands before submission.

**Examples:**

| Input | Transformed To |
|-------|---------------|
| `l` | `look` |
| `k goblin` | `kill goblin` |
| `cast ${target}` | `cast goblin` (if target=goblin) |

#### Alias Processing Pipeline

> **Note:** See Section 7.1 Unified Execution Model for the authoritative processing pipeline.eterministic behavior regardless of storage order.

#### Alias Types

| Type | Description | Example |
|------|-------------|---------|
| `exact` | Full command match | `l` ? `look` |
| `prefix` | Command prefix with argument forwarding | `k` ? `kill {args}` |

#### Alias Data Structure

```json
{
  "items": [
    {
      "id": "uuid",
      "pattern": "l",
      "type": "exact",
      "replacement": "look",
      "enabled": true
    },
    {
      "id": "uuid",
      "pattern": "k",
      "type": "prefix",
      "replacement": "kill {args}",
      "enabled": true
    }
  ]
}
```

**Pattern Rules:**
- Exact aliases: pattern must match entire input
- Prefix aliases: pattern is matched against input prefix; remaining input is appended to replacement as `{args}`

### 5.3 Triggers

#### Definition

Triggers monitor incoming MUD output and automatically execute commands when a condition is matched.

**Example:**

| Server Output | Trigger Action |
|---------------|----------------|
| `You are hungry.` | `eat bread` |

#### Trigger Processing Pipeline

```
MUD output
     ?
Trigger engine
     ?
submitCommand()
     ?
WebSocket
     ?
MUD server
```

**Rules:**
- Triggers operate on complete output lines
- **Line normalization:** Automation engine receives `normalizedLines: string[]` after terminal processing (xterm splits MUD stream into lines)
- Trigger actions submit through `submitCommand()` (same pipeline as user input)
- Regex triggers are explicitly excluded from SP05

#### Trigger Types

| Type | Description | Example |
|------|-------------|---------|
| `contains` | Substring match | `"You are hungry"` matches `"You are hungry."` |

#### Trigger Safeguards

To prevent runaway loops:

- **Cooldown interval:** Minimum time between trigger executions (per-trigger setting)
- **Internal expansion loop protection:** Max recursion depth (3) for @alias chains

**Example:**
- Trigger: `"You swing"` ? `"attack"`
- When trigger fires "attack", server outputs "You swing"
- Trigger must NOT re-fire for this output (prevents infinite loop)

**Implementation:**
1. Trigger fires ? store `lastTriggerCommand`
2. When processing server output, ignore first output line matching the trigger's match pattern
3. Clear `lastTriggerCommand` after one cycle

**Default Values:**

| Setting | Default |
|---------|---------|
| `cooldown_ms` | 2000 |
| `max_triggers_per_second` | 10 |

#### Trigger Data Structure

```json
{
  "items": [
    {
      "id": "uuid",
      "match": "You are hungry",
      "type": "contains",
      "action": "eat bread",
      "cooldown_ms": 2000,
      "enabled": true
    }
  ]
}
```

### 5.4 Environment Variables

#### Definition

Environment variables store reusable values used by automation rules.

**Examples:**

| Variable | Value |
|----------|-------|
| `target` | `goblin` |
| `heal_hp` | `40` |

#### Variable Reference Syntax

Variables are referenced using `${variable_name}` syntax in aliases and trigger actions.

**Example:**

- Alias pattern: `cast`
- Alias replacement: `cast ${target}`
- If user types `cast`, it transforms to `cast goblin`

#### Environment Data Structure

```json
{
  "items": [
    {
      "id": "uuid",
      "name": "target",
      "value": "goblin"
    },
    {
      "id": "uuid",
      "name": "heal_hp",
      "value": "40"
    }
  ]
}
```

### 5.5 Automation Execution Model

#### Command Source Context

Commands submitted to the pipeline include a source indicator:

| Source | Description |
|--------|-------------|
| `user` | Direct user input |
| `alias` | Input transformed by alias |
| `trigger` | Action fired by trigger |

The source context determines whether alias evaluation occurs:

- **user source:** Alias engine evaluates the command
- **alias source:** Alias engine does NOT re-evaluate (prevents recursion)
- **trigger source:** Alias engine does NOT evaluate (triggers submit directly)

#### Trigger Command Execution

Trigger actions are treated as **direct MUD commands** by default:

```
Trigger action
     ?
submitCommand(source="trigger")
     ?
WebSocket
     ?
MUD Server
```

**Rules:**
- Trigger actions do NOT pass through the alias engine
- This prevents unintentional alias recursion and automation loops
- Variable substitution (`${variable}`) still occurs in trigger actions

**Example:**

| Trigger match | Trigger action | Sent to MUD |
|---------------|----------------|-------------|
| `You are hungry` | `eat apple` | `eat apple` |

Aliases are NOT automatically evaluated in trigger actions.

#### Trigger Action Evaluation

Trigger action text may contain reference symbols that are evaluated before submission.

**Reference symbols:**

| Symbol | Meaning |
|--------|---------|
| `${variable}` | Substitute environment variable |
| `@alias` | Explicit alias invocation |
| `@@` | Escape sequence for literal @ |

**Evaluation rules:**

- Trigger actions do not automatically pass through the alias engine
- Trigger actions are evaluated only when reference symbols are present
- If no reference symbols, text is sent verbatim to MUD

**Evaluation order:**

```
Trigger action text
      ?
Variable substitution (${name})
      ?
Explicit alias invocation (@alias args)
      ?
submitCommand(source="trigger")
      ?
WebSocket ? MUD server
```

**Examples:**

| Trigger Action | Sent to MUD |
|----------------|-------------|
| `eat bread` | `eat bread` |
| `kill ${target}` (target=goblin) | `kill goblin` |
| `@eat` (alias eat=eat bread) | `eat bread` |

#### Explicit Alias Invocation

Aliases may be explicitly invoked from triggers or other contexts using the `@` prefix:

```
@alias_name arguments
```

**Example:**

Trigger action:
```
@eat apple
```

If alias `eat` exists with replacement `eat ${args}`, the alias engine expands it:
```
eat apple
```

This allows automation reuse without enabling implicit recursion.

#### Variable Reference Syntax

Variables are referenced using `${variable_name}` syntax:

| Syntax | Description |
|--------|-------------|
| `${variable_name}` | Substitute with variable value |

**Example:**

- Variable: `target` = `goblin`
- Alias replacement: `cast ${target}`
- User input: `cast`
- Transformed: `cast goblin`

**Variable Substitution Pipeline:**

```
User input
     ↓
Command Separation (semicolon)
     ↓
Command Queue (left-to-right)
     ↓
Per-Command Evaluation:
     ├ Variable substitution (${name} → value)
     ├ Alias evaluation (when allowed)
     └ Trigger action expansion
     ↓
submitCommand() per command
     ↓
WebSocket
```

Variable substitution occurs BEFORE alias evaluation **within each command**.

> **Note:** See Section 7.1 Unified Execution Model for the authoritative processing pipeline.

**Command Separation:**

Multiple commands can be entered on a single line using semicolon (`;`) separator.

```
look;inv;draw sword
```

This submits three separate commands: `look`, `inv`, `draw sword`

**Processing Order:**

Command separation occurs FIRST, before any other automation processing:

1. User Input
2. Command Separation (semicolon)
3. Command Queue (left-to-right)
4. Per-Command Evaluation:
   - Variable substitution
   - Alias evaluation (when allowed)
   - Trigger action expansion
5. submitCommand() per command
6. WebSocket → MUD

> **Note:** See Section 7.1 Unified Execution Model for the authoritative processing pipeline.

**Whitespace Handling:**

- Trim leading/trailing whitespace
- Ignore empty commands

**Example:**

```
look;;inv ; ; draw sword
```

Produces:
- `look`
- `inv`
- `draw sword`

**Alias Interaction:**

Command separation occurs FIRST, then aliases expand within each command:

```
k goblin;score;drink potion
```

With alias `k ? kill {args}`:

```
kill goblin;score;drink potion
```

Produces:
- `kill goblin`
- `score`
- `drink potion`

**Trigger Interaction:**

Trigger actions also support command separation:

```
stand;draw sword;kill goblin
```

Produces:
- `stand`
- `draw sword`
- `kill goblin`

**Escaping:**

Escaping semicolons is NOT supported in SP05. Semicolons always act as separators.
- User input (before alias evaluation)
- Alias replacements
- Trigger actions

#### Automation Recursion Protection

The automation engine enforces a maximum recursion depth:

| Limit | Value |
|-------|-------|
| `max_recursion_depth` | 3 |

If recursion depth exceeds the limit, automation execution stops.

**Loop Avoidance (Internal Only)**

The client must not prevent or dampen trigger execution based on repeated MUD output. Triggers may fire indefinitely if the MUD continues to emit matching lines.

Loop avoidance applies only to internal automation expansion where rules reference other rules (e.g., explicit alias invocation via '@') or where command chaining (';') expands command queues without a new user input or new MUD output event.

The automation dispatcher enforces:
- A maximum expansion depth (`max_recursion_depth`) for alias invocation chains.
- A maximum number of commands emitted by a single dispatch (`max_commands_per_dispatch`), where a "dispatch" is processing one user-entered line or one trigger action string.
- Cycle detection for explicit alias invocation chains using a visited set of alias IDs.

**Dispatch Cycle Definition:**

- **User dispatch:** processing one line of user input (which may contain ';')
- **Trigger dispatch:** processing one trigger action string (which may contain ';')

Internal loop protection applies within a single dispatch cycle only.

#### Reference Symbol Escaping

The `@` character may appear in normal MUD commands. Use `@@` to send a literal `@`:

| Input | Sent to MUD |
|-------|-------------|
| `@@tell hello` | `@tell hello` |

This ensures players can send commands beginning with `@` when desired.

#### Behavior Summary

| Input | Behavior |
|-------|----------|
| `eat apple` | Direct command (alias evaluated) |
| `@eat apple` | Invoke alias `eat` with args |
| `${target}` | Variable substitution |
| `@@` | Literal `@` |

## 6. Implementation Notes

Automation state should be loaded with the profile when connecting:

1. User connects to MUD
2. Fetch connection profile (includes aliases, triggers, variables)
3. Load automation engine with profile data
4. Enable alias processing on input
5. Enable trigger processing on output

### 6.2 Disconnection Handling

When session disconnects:
- Disable all trigger processing
- Clear trigger cooldown timers
- Keep alias/variable data in memory for reconnection

### 6.3 Profile Update Behavior

If user edits automation while connected:
- Apply changes immediately to in-memory engine
- Persist changes to server in background

### 6.4 Automation Dispatcher Architecture

Implement a single command dispatcher instead of scattered processing functions:

```
processCommand(source: string, text: string)
    // source: "user" | "alias" | "trigger"
    
    if source == "user":
        runVariableSubstitution(text)
        runAliasEngine(text)
    else if source == "alias":
        // No alias re-evaluation (prevents recursion)
        runVariableSubstitution(text)
    else if source == "trigger":
        runVariableSubstitution(text)
        runExplicitAliasInvocation(text)
    
    runCommandSeparation(text)
    
    for cmd in commands:
        submitCommand(cmd)
```

**Benefits:**
- Consistent logic across all command sources
- Prevents recursion bugs
- Easier future scripting integration
- Timers and macros plug into the same dispatcher
- Source parameter controls alias evaluation behavior

## 7.1 Unified Execution Model

> **Note:** This is the authoritative execution model. If other sections describe different behavior, this model takes precedence.

The automation system follows this deterministic execution model:

```
User Input
   ?
Command Separation (semicolon)
   ?
Command Queue (left-to-right)
   ?
Per-Command Evaluation:
   � Variable substitution
   � Alias evaluation (when allowed)
   � Trigger action expansion
   ?
Nested commands inserted into queue
   ?
submitCommand()
```

**1. Command Separation Happens First**

When a command line is entered, the client first splits on semicolon. This is the correct MUD client model - it avoids alias engines having to parse semicolons themselves.

```
k goblin;score;drink potion
```

Becomes command queue:
- `k goblin`
- `score`
- `drink potion`

**2. Sequential Depth-First Execution**

Commands execute left-to-right. If any command produces additional commands (alias expansion or trigger action), those execute immediately before continuing.

**Example:**
- User input: `k goblin;score`
- Alias: `k ? kill %1;stance offensive`
- Execution order:
  1. `kill goblin`
  2. `stance offensive`
  3. `score`

**3. Nested Command Expansion**

If alias/trigger actions generate commands with semicolons, those are also split and executed sequentially.

**4. No Artificial Command Count Limits**

No arbitrary limits on command chain length. The system must not reject command chains solely due to length.

Example that must remain valid:

```
look;inv;draw sword;stance offensive;attack goblin
```

Or large alias expansions that produce many commands.

**5. Required Loop Protection**

- **Recursion Depth:** Max 3
- **Trigger Throttle:** Max 10/sec
- **Automation Loop Detection:** Detect repeated command signatures during dispatch; halt if infinite loop detected

**6. Command Queue with Dispatch Backpressure**

Command execution uses a client-side command queue rather than immediately sending all commands to the MUD.

```
User Input / Trigger Action
        ?
Command Separation
        ?
Command Queue
        ?
Sequential Dispatch
        ?
submitCommand()
        ?
WebSocket ? MUD Server
```

**Execution Rules:**

- Commands are processed sequentially from the queue
- Only a small number of commands may be in-flight at a time
- The dispatcher advances when either server output is received, OR a fallback timeout occurs
- This prevents the client from flooding the server while still allowing arbitrarily long command chains

**7. Loop Detection Instead of Hard Limits**

Automation safety relies on loop detection, not command count limits.

The automation engine tracks a rolling history of recently dispatched commands. Each dispatch event records:

| Field | Description |
|-------|-------------|
| `source` | user | alias | trigger |
| `normalized_command` | The command text after all transformations |
| `timestamp` | When the command was dispatched |
| `trigger_id` | ID of trigger that caused this (if applicable) |
| `alias_id` | ID of alias that caused this (if applicable) |

**Loop Detection Rule:**

If the same command signature repeats rapidly without user input, the system detects a potential loop.

Example pattern:
```
attack goblin
attack goblin
attack goblin
attack goblin
...
```

If repeated beyond a reasonable threshold in a short time window, automation is paused.

**8. Trigger Self-Activation Protection**

Trigger loops must be prevented when triggers cause commands that produce matching output.

Instead of ignoring a single output line, triggers suppress self-activation during the same dispatch cycle.

**Example Scenario:**
- Trigger: `"You swing"` ? `attack`
- When the trigger fires and sends attack, any output generated during that same dispatch cycle must not re-trigger the same automation rule

**Implementation:**
1. Trigger fires ? store `lastTriggerCommand` with dispatch cycle ID
2. When processing server output, track cycle ID
3. Ignore trigger matches that would fire for output from the same cycle
4. Clear `lastTriggerCommand` after cycle completes

**9. Automation Circuit Breaker**

When loop detection identifies runaway automation:

1. Automation dispatch pauses immediately
2. The command queue stops processing
3. The UI notifies the user:

```
Automation paused: possible loop detected.
Review triggers or aliases before resuming.
```

**User Options:**
- Disable triggers
- Disable the offending rule
- Resume automation manually

This prevents infinite loops without introducing artificial limits.

**10. Memory Safety Considerations**

While command count limits are intentionally avoided, the implementation ensures the client does not allocate unbounded memory.

**Recommended Safeguards:**
- Streaming command parsing (avoid storing massive command strings)
- Efficient queue structures with bounded size
- Loop detection before unbounded growth occurs

These protections remain implementation-level safeguards rather than user-visible limits.
