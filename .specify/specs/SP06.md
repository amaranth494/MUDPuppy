# SP06 — Documentation & UX Refinement

> **Spec ID:** SP06  
> **Status:** Draft  
> **Author:** Product Lead  
> **Branch Prefix:** sp06-docs-ux-refinement  
> **Constitutional Reference:** Constitution II (User Experience), Constitution III (Phased Development)  

---

## 1. Problem Statement

The current MUDPuppy interface provides core functionality (connections, profiles, automation) but lacks the usability infrastructure needed for users to discover, understand, and effectively use these features. The Help system exists in name only, contextual guidance is absent, and several UX friction points hinder user adoption.

SP06 addresses these gaps by building a first-class Help system, adding contextual guidance throughout the UI, and refining key interaction patterns. This spec intentionally does not introduce new automation capabilities—those introduced in SP03–SP05 must become understandable and accessible to users.

---

## 2. Objectives

SP06 must deliver:

1. **Structured Help System** — Transform the placeholder Help page into comprehensive documentation covering all user-facing features
2. **Contextual Inline Guidance** — Add helper text and tooltips throughout the UI, especially in automation editors
3. **Improved Navigation** — Make Connection Settings discoverable from the main Play screen when connected
4. **Automation Templates** — Provide copy-ready examples for common automation patterns
5. **Ordering Controls** — Enable users to reorder aliases and triggers by priority
6. **Automation Status UI** — Clearly communicate when automation safety systems are active
7. **Command History** — Implement up/down arrow navigation for command recall
8. **Help Search** — Add quick search capability to jump to relevant documentation

SP06 does NOT introduce:

- Scripting languages
- Regex triggers
- Timers
- Conditional automation
- Automation import/export
- Trigger chains or nesting

> **This is a usability and documentation spec, not an automation expansion spec.**

---

## 3. Non-Goals (Explicitly Out of Scope)

SP06 will NOT include:

- Client-side scripting or scripting language support
- Regex-based trigger matching
- Timer-based automation
- Conditional logic in aliases or triggers
- Automation import/export functionality
- Automation sharing between users
- Multi-command trigger actions
- Variable expressions or math
- Output suppression ("gagging")
- Trigger chains or nesting

> **All automation capabilities beyond SP05 remain out of scope for this spec.**

## 4. Help System Specification

### 4.1 Structure

The Help system shall contain the following top-level sections:

```
Help
 ├── Getting Started
 ├── Connecting to a MUD
 ├── Terminal Controls
 ├── Key Bindings
 ├── Aliases
 ├── Triggers
 ├── Environment Variables
 ├── Automation Safety
 └── Troubleshooting
```

### 4.2 Getting Started Section

The Getting Started section shall include:

- Account creation and login overview
- Quick Connect workflow (host:port → connect)
- Saved Connections workflow (create → save → connect)
- Navigating the main interface (Play, Connections, Help tabs)

### 4.3 Connecting to a MUD Section

The Connecting to a MUD section shall include:

- Explanation of host/port configuration
- SSL/TLS connection options
- Saved vs. Quick Connect comparison
- Connection timeout handling
- Reconnection behavior

### 4.4 Terminal Controls Section

The Terminal Controls section shall include:

- Input behavior (echo, command submission)
- Scrollback behavior and buffer management
- Copy/paste behavior (Ctrl+C, Ctrl+V, mouse selection)
- Timestamp display options
- Word wrap configuration
- Clear screen commands

### 4.5 Key Bindings Section

The Key Bindings section shall explain:

- How keybindings translate keystrokes into commands
- That keybindings update live as mappings change
- Common default mappings
- Custom binding creation workflow

Example mappings to document:
- `F1` → `score`
- `Ctrl+F1` → `help`

### 4.6 Aliases Section

The Aliases section shall explain:

- Alias pattern definition
- Alias types: **exact** (full match) vs. **prefix** (partial match)
- Replacement text with `{args}` placeholder
- Command chaining using `;` separator

Examples to document:

| Pattern | Type | Replacement | Effect |
|---------|------|-------------|--------|
| `l` | exact | `look` | `l` expands to `look` |
| `k` | prefix | `kill {args}` | `k goblin` expands to `kill goblin` |
| `k` | prefix | `kill {args};stance offensive` | Chains commands |

### 4.7 Triggers Section

The Triggers section shall explain:

- Substring matching (case-sensitive, currently)
- Cooldown periods and their purpose
- Trigger actions (single command execution)
- Why triggers execute client-side only

Example to document:
- Match: `You are hungry`
- Action: `eat bread`
- Cooldown: `5000` (5 seconds)

### 4.8 Environment Variables Section

The Environment Variables section shall explain:

- Variable syntax: `${variable_name}`
- Creating and editing variables
- Using variables in aliases and triggers
- Variable persistence

Example workflow:
1. Create variable: `target = goblin`
2. Use in alias: `k → kill ${target}`

### 4.9 Automation Safety Section

The Automation Safety section shall explain the protections introduced in SP05:

- **Recursion limits** — Maximum alias expansion depth (3 levels)
- **Command queue** — Maximum commands per dispatch (200)
- **Trigger cooldown** — Minimum time between trigger firings
- **Circuit breaker** — Automatic pause when automation loops detected

This section must clearly communicate why automation may pause and how to resume it.

### 4.10 Troubleshooting Section

The Troubleshooting section shall include:

- Common connection issues
- Automation not firing (check: enabled, pattern, cooldown)
- Connection drops and reconnection
- How to report bugs

## 5. Contextual Help Specification

### 5.1 Alias Editor Guidance

The Alias Editor UI shall include helper text:

> **Aliases transform typed commands before they are sent to the server.**
>
> Example:
> - `l` → `look`
> - `k goblin` → `kill goblin`

This text shall appear:

- Above the alias list
- In a tooltip on first alias creation

### 5.2 Trigger Editor Guidance

The Trigger Editor UI shall include a note:

> **Triggers activate when matching server output is received.**
>
> Matching is currently case-sensitive substring matching.

This text shall appear:

- Above the trigger list
- As a tooltip in the match pattern field

### 5.3 Environment Variables Guidance

The Environment Variables panel shall include explanation:

> **Variables can be referenced using ${variable_name}.**
>
> Example: `cast ${target}`

This shall appear:

- Above the variables list
- As inline help in the variable editor

### 5.4 Keybinding Editor Guidance

The Keybinding Editor shall include:

- Description of what the keybinding will send
- Warning if the keybinding conflicts with browser shortcuts

## 6. Connection Settings Navigation

### 6.1 Current State (SP05)

Currently, Connection Settings are accessed through:

```
Connections → Edit → Settings
```

This workflow is not discoverable during active play.

### 6.2 SP06 Improvement

When a session is connected, the main navigation shall include:

```
Play
Connections
Connection Settings
Help
```

**Connection Settings** shall:

- Only appear in the nav when actively connected
- Automatically load settings for the active connection
- Display the connection name in the page header

> **This makes configuration discoverable without leaving the Play context.**

## 7. Automation Templates

### 7.1 Template Library

SP06 shall include a built-in library of automation templates stored as TypeScript (`src/templates.ts`):

```typescript
export const aliasTemplates = [
  { pattern: 'k', type: 'prefix', replacement: 'kill {args};stance offensive' },
];

export const triggerTemplates = [
  { match: 'You are hungry', action: 'eat bread', cooldown: 5000 },
];
```

> **Note:** Using TypeScript instead of JSON for better type safety and easier UI integration.

#### Combat Alias Template
```
Pattern: k
Type: prefix
Replacement: kill {args};stance offensive
```

#### Auto-Eat Trigger Template
```
Match: You are hungry
Action: eat bread
Cooldown: 5000
```

#### Healing Variable Template
```
Name: heal
Value: potion
```

Then alias:
```
Pattern: dr
Type: exact
Replacement: drink ${heal}
```

### 7.2 Template Access

Templates shall be accessible via:

- "Add Example" button in Alias Editor
- "Add Example" button in Trigger Editor
- "Examples" section in Help

> **Single source of truth:** All templates live in `src/templates.ts`. The Help Examples section imports from this same module to prevent drift.

### 7.3 One-Click Import

Users shall be able to import a template with a single click, populating the editor with the template values.

## 8. Ordering UX

### 8.1 Reorder Controls

The Alias and Trigger lists shall include:

- **Move Up** button (↑)
- **Move Down** button (↓)

Alternatively, drag-and-drop ordering if technically feasible within the sprint.

> **Execution priority:** Array order determines execution order. Items at index 0 execute first.
> Reordering simply rewrites the array.

### 8.2 Priority Indication

Each alias and trigger shall display its priority order number (1, 2, 3...).

### 8.3 Save on Reorder

Reordering shall persist immediately to the server—no separate save action required.

## 9. Automation Status Indicators

### 9.1 Circuit Breaker UI

When automation pauses due to loop detection or other safety triggers, the UI shall display:

```
Automation paused due to loop detection.
[Resume Automation] [Disable Automation]
```

### 9.2 Status Indicator Location

This message shall appear:

- In a banner at the top of the Play screen
- In the Connection Settings automation panel
- With a warning icon in the sidebar

### 9.3 Resume Behavior

- **Resume Automation** — Clears the circuit breaker, re-enables triggers and aliases
- **Disable Automation** — Turns off all automation for this connection (persists)

## 10. Command History

### 10.1 Basic Recall

Pressing **Up Arrow** shall recall the previous command.
Pressing **Down Arrow** shall recall the next command (if going back in history).

### 10.2 History Scope

- Command history is per-connection
- History is retained during the session
- History is NOT persisted across sessions (SP06 scope)

### 10.3 Empty State

If history is empty, up/down arrow shall have no effect (no errors).

### 10.4 Recalled Commands Don't Duplicate

If a user recalls a command (via up arrow) but does not execute it (does not press Enter), the recalled command shall NOT be added to history again.

Example:
```
History: ["look", "north"]
User presses up → "north" appears in input
User presses Enter → History: ["look", "north"] (no change)

User presses up → "north" appears in input
User presses Enter → History: ["look", "north"] (no change)

User presses up → "look" appears in input
User presses Enter → History: ["look", "north", "look"] (added)
```

> **Rationale:** Only executed commands (submitted via Enter) grow history.

## 13. Help System Architecture

### 13.1 Architectural Principle

> **Help content is platform-owned structured product documentation.**
> 
> In SP06 it is stored in repo-managed files and published via read-only versioned API endpoints.
> The frontend is a consumer of that API, not the sole owner of help content presentation.

This approach preserves future flexibility without overbuilding. The contract between frontend and backend can evolve independently.

### 13.2 Content Model

Help content is stored as structured JSON files in the repository:

```
/help/
  getting-started.json
  connecting.json
  terminal.json
  keybindings.json
  aliases.json
  triggers.json
  variables.json
  safety.json
  troubleshooting.json
```

Each file contains:

```json
{
  "slug": "aliases",
  "title": "Aliases",
  "description": "Transform typed commands before sending to server",
  "content": "...markdown or structured content...",
  "examples": [...]
}
```

### 13.3 API Surface

| Endpoint | Description |
|----------|-------------|
| `GET /api/v1/help` | List all help sections (slug, title, description) |
| `GET /api/v1/help/:slug` | Get full help content for a section |

**Note:** Search is client-side. Frontend fetches all content once via `/help`, then filters locally.

**Constraints:**
- No database storage
- No authoring UI
- No write endpoints
- No auth-dependent personalization
- No plugin execution model
- Search performed client-side

### 13.4 Reorder API

Aliases and triggers are stored as JSONB arrays within the profile. Reordering should use the existing PUT endpoints:

- `PUT /profiles/{id}/aliases` — send entire reordered array
- `PUT /profiles/{id}/triggers` — send entire reordered array

No new endpoints required.

### 13.5 Automation Status

Per Constitution I, automation runs server-side. Circuit breaker state must also be server-side for proper enforcement and observability.

| Endpoint | Description |
|----------|-------------|
| `GET /api/v1/sessions/:id/automation/status` | Get automation status (paused, reason, canResume) |
| `POST /api/v1/sessions/:id/automation/resume` | Resume paused automation |
| `POST /api/v1/sessions/:id/automation/disable` | Disable automation for session |

Frontend reads status from API and displays in UI.

## 14. Data Model Changes

### 14.1 Alias JSONB Objects

Aliases are stored as a JSONB array in the profile. Execution order is determined by array position—index 0 executes first.

```json
{
  "items": [
    { "id": "...", "pattern": "k", "replacement": "kill {args}", "enabled": true }
  ]
}
```

> **No order field.** Reordering simply rewrites the array.

### 14.2 Trigger JSONB Objects

Triggers are stored as a JSONB array in the profile. Execution order is determined by array position—index 0 executes first.

> **No order field.** Reordering simply rewrites the array.
